# Architecture — gymbro

A single-file Flask app plus a daily cron image generator. No database. No build step. No auth layer (relies on Tailscale VPN for trust).

## High-level

```
┌─────────────┐       HTTPS       ┌──────────────────────┐
│  iPhone PWA │ ────────────────► │  Flask :7000         │
│  (Tailscale)│                   │  /home/work/gym_web.py│
└─────────────┘                   └──────────┬───────────┘
                                             │
                                             ▼
                                ┌────────────────────────┐
                                │ .whoop_workout_log.json│
                                │ .hermes/image_cache/   │
                                └────────────────────────┘
                                             ▲
                                             │
                                ┌────────────┴───────────┐
                                │ gymbro_daily_image.py  │
                                │ cron @ 06:00 HKT       │
                                └────────────────────────┘
```

## API endpoints

| Method | Path | Purpose |
|---|---|---|
| `GET`  | `/`                   | Render the single-page UI (HTML embedded in `gym_web.py`) |
| `GET`  | `/healthz`            | Liveness probe — `{"status":"ok","time":...,"today":...}` |
| `GET`  | `/api/state`          | Full current session (today's exercises, start_time, etc.) |
| `POST` | `/api/log_set`        | Log one set — `{exercise, weight_kg, reps, set, intensity, form_check}` |
| `POST` | `/api/finish_exercise`| Mark exercise done, returns set-breakdown + volume summary |
| `POST` | `/api/end_session`    | Finalize workout, mark `completed=true`, write `end_time` |
| `GET`  | `/api/today_image`    | Returns metadata for today's cached motivation image |
| `GET`  | `/api/streak`         | Consecutive workout-day count |
| `GET`  | `/img/<filename>`     | Static serve from `/home/work/.hermes/image_cache/` |

## Data persistence

### Workout log

- File: `/home/work/.whoop_workout_log.json`
- Shape: `{ "<YYYY-MM-DD>": { date, start_time, end_time, completed, context, exercises: [...] } }`
- Writes are full-file rewrites (`json.dumps(..., indent=2, ensure_ascii=False)`) — fine at <10 sets/day, would need rewriting at scale
- Lock-free — single-user (Jim)

### Per-set entry

```json
{
  "exercise": "Bench Press",
  "weight_kg": 60,
  "weight_note": "felt easy",
  "reps": 10,
  "set": 3,
  "intensity": "working",
  "form_check": "ok",
  "time": "2026-07-19T18:42:11+08:00",
  "source": "gym-web-tap"
}
```

## Image cache

- Path: `/home/work/.hermes/image_cache/gymbro_<YYYY-MM-DD>.png`
- Generated by `scripts/gymbro_daily_image.py` once per morning
- Served by Flask via `/img/<filename>` (the app's `static_folder` is set to this directory)
- 1-day cache TTL — re-uses today's file if it exists

## Default reps and intensity

- **Default reps** = 10 (calibrated to Jim's training plan; sets `default_reps()` in `gym_web.py`)
- **Intensity auto-detection** (`detect_intensity(set_n)`):
  - set 1 → `warm-up`
  - set 2 → `warm-up`
  - sets 3-4 → `working` (target 4 working sets)
  - sets 5+ → `burn-out`
- **Workout share ratio**: ~60% compounds / 40% accessories (Jim's split)

## Daily image generation

`scripts/gymbro_daily_image.py` calls the `MiniMax image-01` model:

```
POST https://api.minimax.io/v1/image_generation
{
  "model": "image-01",
  "prompt": "motivational gym scene, ...",
  "width": 1024,
  "height": 1024,
  "response_format": "url"
}
```

Downloads the returned `image_urls[0]` and writes to the cache path. Pre-flight `probe_image_endpoint.py` is recommended before first run.

## Why no DB?

- Single user → write contention is zero
- File is human-readable → `cat .whoop_workout_log.json | jq` works
- Git-friendly → Jim can `git log -p` his training history if he wants to
- Trivial to back up (`cp` to NAS)

If this ever became multi-user or hit >100 sets/day, the right move is SQLite + WAL mode, not Postgres.

## Why no auth?

- Bound to Tailscale IP `100.114.66.125` — only devices on Jim's tailnet can route to it
- Tailnet is treated as a trusted network (HTTPS via Tailscale certs)
- No public exposure = no attack surface worth mitigating with login UI

## Deployment

There's no production deployment pipeline. The process runs on Jim's home server under `tmux`/`screen` or whatever's already running his other Flask services. CI (`.github/workflows/python-ci.yml`) only checks syntax — actual runs are manual.
